# Pragmas: Monitoring and Visibility

These conventions apply to all AUT libraries written in IEC 61131-3 Structured Text.

---

## `{attribute 'monitoring' := 'call'}`

On every monitoring refresh (when the online view is open), TwinCAT calls the GET accessor of the property to read the current value.

**Use when:**
- The property is GET-only (no SET accessor exists)
- The value is computed dynamically (e.g. `ADR(THIS^)`, counters, state machines)
- The backing variable is written directly inside the FB body, not through a SET accessor
- The GET accessor has no side effects — the call cost is negligible

**Note on read-only properties in ST:** Structured Text has no equivalent of C#'s `public get; private set;`. A read-only property has only a GET accessor. Internally the FB writes directly to the backing variable (`_value := x`); calling `THIS^.Prop := x` is not possible because no SET exists. For this reason, `'call'` is the only correct choice for read-only properties.

```
{attribute 'monitoring' := 'call'}
PROPERTY InstancePath : STRING(1024)
```

Examples from AUT_Core: `InstancePath`, `IsValid`, `Identity`, `Path`, `Parent`, `Root` all use `'call'`.

---

## `{attribute 'monitoring' := 'variable'}`

TwinCAT creates an internal hidden variable for the property. This variable is updated **each time the PLC program itself calls the GET or SET** accessor. The monitoring view reads the hidden variable directly, without executing any code.

**Use when:**
- The property has both a public GET and a public SET
- The value is always assigned through the SET accessor — never by writing directly to the backing variable inside the FB
- The GET is called frequently by the PLC program (not only by the monitoring system)

**Limitations:**

| Case | Result |
|---|---|
| GET-only property; backing variable written in `FB_Init` | Hidden variable never updated — monitoring shows the default value |
| Value written directly to `_backing` inside the FB | Hidden variable not updated — monitoring shows a stale value |
| Property with GET+SET; value always set via SET by callers | Works correctly |

The root cause: TwinCAT injects an update of the hidden variable at each GET or SET call site generated by the PLC program. If no SET exists and the GET is never called by the program, the hidden variable is never written.

```
{attribute 'monitoring' := 'variable'}
PROPERTY DelayOnTime : TIME
```

Examples from AUT_Core: `DelayOnTime` and `DelayOffTime` on `IDigitalSignal` use `'variable'` because they are configuration parameters exposed with GET+SET and always set by callers through the SET accessor.

## Where to place monitoring pragmas

The pragma takes effect on the **type through which the object is accessed** in the online view.

| Access via | Relevant pragma location |
|---|---|
| Interface reference (`myRef : IMyDevice`) | Interface declaration |
| Concrete FB variable (`myCyl : PneumaticCylinder`) | FB implementation |

**Rule: place the pragma in both the interface and the implementation.** There is no conflict, and both access patterns are covered.

```
// In interface IDigitalSignal
{attribute 'monitoring' := 'call'}
PROPERTY Value : BOOL

// In FB DigitalSignal
{attribute 'monitoring' := 'call'}
PROPERTY Value : BOOL
```

**In derived classes:** the pragma is inherited only when the property is not overridden. If a derived FB overrides a property, the pragma must be repeated in the derived FB's property declaration.

---

## `{attribute 'displaymode' := 'hex'}`

Controls the numeric display format of a variable or property in the monitoring view. Overrides the project-wide global setting.

**Valid values:**

| Value | Format |
|---|---|
| `'hex'` or `'hexadecimal'` | Hexadecimal |
| `'bin'` or `'binary'` | Binary |
| `'dec'` or `'decimal'` | Decimal |

The two pragmas combine independently: `displaymode` controls the format; `monitoring` controls how the value is acquired.

```
{attribute 'displaymode':='hex'}
{attribute 'monitoring' := 'call'}
PROPERTY Identity : ULINT
```

Example from AUT_Core: `Identity` (a runtime memory address) uses `'hex'` because addresses are naturally read in hexadecimal.

---

## `{attribute 'hide'}`

Hides a single variable from the online view, the input assistant, and cross-reference search. The variable remains accessible inside the FB but:

- generates no ADS symbol
- does not appear in I/O mapping
- cannot be made persistent
- cannot have breakpoints or watch expressions applied to it

```
{attribute 'hide'}
_internalState : INT;
```

Use this on individual backing variables that are fully exposed through a property with a monitoring pragma.

---

## `{attribute 'hide_all_locals'}`

Equivalent to applying `{attribute 'hide'}` to every local variable in a POU in a single declaration. It must appear before the `VAR` block.

```
{attribute 'reflection'}
{attribute 'hide_all_locals'}
FUNCTION_BLOCK Object IMPLEMENTS IObject
VAR
    ...
END_VAR
```

**Difference from `'hide'`:** `'hide'` targets one variable; `'hide_all_locals'` targets the entire local scope of the POU.

**Caution:** with `'hide_all_locals'` active, local variables are not visible during debugging. Use only when the FB exposes everything relevant through properties with monitoring pragmas configured.

Examples from AUT_Core: `Object` and `DigitalSignal` use `'hide_all_locals'` because all observable state is exposed through properties with `monitoring := 'call'`.

---

## Decision guide

```
Read-only property (GET only)?
  -> 'call'  [mandatory — no SET exists to update the hidden variable]

Property with GET+SET, value always set via SET by callers?
  -> 'variable'  [monitoring reads memory, no code executed]

Value computed dynamically or written directly to backing variable internally?
  -> 'call'

Numeric value to display in a specific format?
  -> add 'displaymode'

Internal variable that should not appear in online view?
  -> 'hide' (single variable) or 'hide_all_locals' (entire POU scope)
```
